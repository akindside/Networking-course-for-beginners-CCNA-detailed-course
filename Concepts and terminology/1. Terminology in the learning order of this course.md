###### Probe
In networking and cybersecurity, a **probe** is a specialized packet or signal sent to a target system to elicit a response that reveals information about its status, open ports, or active services.

###### Hardcoded
In a penetration testing (pentesting) context,
hardcoded refers to the practice of embedding sensitive information—such as credentials, API keys, encryption secrets, or configuration values—directly into the application's source code, compiled binaries, or configuration files, rather than obtaining them dynamically at runtime.

###### Certificate transparency log
Certificate Transparency (CT) is an open framework and public auditing system designed to prevent the fraudulent issuance of SSL/TLS certificates by requiring all Certificate Authorities (CAs) to log every issued certificate in publicly accessible, append-only databases. When a CA issues a certificate, it must submit it to multiple independent CT logs to receive a Signed Certificate Timestamp (SCT), which is then embedded in the certificate and verified by modern web browsers; if a certificate lacks this proof of logging, the browser will flag the connection as untrusted. From a cybersecurity and reconnaissance perspective, CT logs are a "gold mine" for penetration testers because they provide a permanent, searchable record of every subdomain ever registered for a target organization, often revealing hidden development, staging, or internal servers (e.g., dev-vpn.target.com) that would otherwise remain invisible during standard DNS brute-forcing.

###### Web application
A web application is a software program stored on a remote server and delivered over the internet through a browser interface, utilizing a client-server architecture to perform specific tasks for the user. Unlike static websites that only display information, web applications are interactive, processing user input through a front-end (typically HTML, CSS, and JavaScript) and a back-end (languages like Python, PHP, or C) that communicates with a database to retrieve or store data. From a penetration testing perspective, web applications represent a significant attack surface because they are publicly accessible and often contain vulnerabilities—such as SQL Injection, Cross-Site Scripting (XSS), or broken authentication—that can be exploited to gain unauthorized access to sensitive user information or internal server resources.

###### Registration record
Domain registration records, primarily accessed via the WHOIS and RDAP protocols, act as the "land registry" of the internet, providing a structured profile of a domain’s owner, registrar, and technical configuration. For a penetration tester, these records are essential for mapping an organization's attack surface because they reveal critical data points like name servers (pointing to the DNS infrastructure), registration and expiration dates (identifying neglected or vulnerable assets), and administrative contact information that can be leveraged for social engineering or identifying related domains through "reverse WHOIS" searches. Although modern privacy regulations like GDPR have led to the redaction of many personal details, the transition from the plain-text WHOIS to the structured, JSON-based RDAP (Registration Data Access Protocol) has improved the ability of security tools to programmatically audit these records for "red flags," such as a domain recently registered with a high-risk registrar or one missing its SCT (Signed Certificate Timestamp) in the public log.

###### SSL/TLS certificate
An SSL/TLS certificate is a digital credential that authenticates a website's identity and enables an encrypted connection between a web server and a browser using the Transport Layer Security (TLS) protocol. Issued by a trusted Certificate Authority (CA), the certificate contains the domain name, the owner's identity, and the server's public key, which is used during the initial TLS Handshake to negotiate a unique session key for data encryption. For a penetration tester, examining these certificates during reconnaissance—via tools like OpenSSL, Censys, or crt.sh—is vital because they often reveal a wealth of information, including valid subdomains (via the Subject Alternative Name field), the underlying server infrastructure, and potential vulnerabilities if the certificate is self-signed, expired, or uses outdated cryptographic algorithms like SHA-1.

###### Self signed Certificate
an identity certificate created, signed, and owned by the creator rather than a trusted Certificate Authority (CA)

###### AWS credentials
AWS credentials are the digital authentication keys—typically consisting of an **Access Key ID** and a **Secret Access Key**—that grant programmatic access to Amazon Web Services resources. For a penetration tester, these credentials are high-value targets often discovered during reconnaissance by scanning public GitHub repositories for hardcoded strings, probing for **Server-Side Request Forgery (SSRF)** vulnerabilities to access the **Instance Metadata Service (IMDS)**, or searching for misconfigured `.env` files. Because these keys can provide a direct path to sensitive data or administrative control over cloud infrastructure, security best practices mandate using temporary **IAM Roles** instead of long-term static keys and enforcing the principle of **Least Privilege** to minimize the impact of a potential leak.

###### Metadata
Metadata is "data about data"—a hidden layer of information embedded within a file that describes its properties, origins, and characteristics without being part of the primary content. While a photo shows an image, its metadata (often in EXIF format) reveals the exact GPS coordinates, camera model, and timestamp of when it was taken; similarly, a Word document's metadata contains the author's name, total editing time, and the local file path where it was saved. For a penetration tester, harvesting metadata using tools like ExifTool or FOCA is a critical passive reconnaissance step, as it can leak internal usernames, software versions, and network naming conventions that help craft highly targeted social engineering attacks or identify vulnerable server-side processing scripts.

###### Patch and patching
Patching is the process of applying a "patch"—a small piece of software or code update—to an existing program, operating system, or database to fix bugs, address security vulnerabilities, or improve performance. In the world of cybersecurity, Security Patching is the most critical defensive measure, as it closes the "holes" that hackers use to gain unauthorized access; once a vulnerability is publicly disclosed (known as a CVE), attackers move quickly to exploit systems that haven't been patched yet. For a penetration tester, identifying "unpatched" systems is the primary goal of the reconnaissance and scanning phases, as these legacy versions often have well-documented exploit scripts available in databases like Exploit-DB, making them the path of least resistance into a network.

###### Autonomous System Number (ASN)
An Autonomous System Number (ASN) is a unique numerical identifier assigned to a large collection of IP networks, known as an Autonomous System, which is managed by a single entity like an ISP, a university, or a major corporation. In the context of reconnaissance, identifying a target's ASN allows a pentester to map out the organization's entire public IP footprint and infrastructure by querying BGP (Border Gateway Protocol) routing tables to see which IP prefixes that specific ASN announces to the internet. This high-level view helps distinguish between assets hosted on-premise and those residing in third-party cloud environments, providing a foundational "blast radius" for further vulnerability scanning and attack surface analysis.

###### SPF record
An SPF (Sender Policy Framework) record is a specific type of DNS TXT record that lists the authorized mail servers allowed to send emails on behalf of a domain. When an email is sent, the receiving mail server checks the SPF record of the "From" domain to verify that the IP address of the sending server matches one of the approved addresses listed in the DNS. If the IP is not on the list, the email may be flagged as spam or rejected entirely, preventing email spoofing and phishing attacks where an attacker tries to impersonate a legitimate organization. For a penetration tester, SPF records are valuable during reconnaissance because they often reveal the internal IP ranges of the company's mail servers, third-party services they use (like SendGrid or Microsoft 365), and potential misconfigurations like a "Soft Fail" (~all) or "Neutral" (?all) policy that might allow for successful phishing simulations.

###### Codebase
A codebase is the complete collection of source code, configuration files, and build scripts used to create and maintain a specific software application or system. Typically managed through a version control system like Git, the codebase serves as the "single source of truth" for developers, encompassing everything from high-level application logic to low-level utility functions and unit tests. In a security context, the codebase is a primary target for Static Application Security Testing (SAST), where tools scan the text files for hardcoded credentials, insecure API calls, or logic flaws before the code is ever compiled or deployed.

###### Disallowed path
A disallowed path is a specific directory or file listed within a website's robots.txt file that the site owner explicitly requests web crawlers and search engine bots not to visit or index. For a penetration tester, these paths are high-value targets during the reconnaissance phase because they often point directly to sensitive or non-public areas of the application, such as /admin/, /config/, /backup/, or /temp/, which the administrator is attempting to hide from public search results. While a Disallow directive is not a security control and does not actually block access, it acts as a roadmap of "interesting" locations that may contain unauthenticated management interfaces, old versions of the codebase, or configuration files that can be further probed during active vulnerability scanning.

###### CORS misconfiguration
A CORS (Cross-Origin Resource Sharing) misconfiguration is a security flaw that occurs when a web server's access control policies are too permissive, allowing unauthorized third-party domains to interact with its resources and steal sensitive user data. The most critical vulnerability arises when a server is configured with Access-Control-Allow-Origin: * or dynamically reflects the Origin header from an incoming request without validation, effectively bypassing the Same-Origin Policy (SOP) that normally prevents scripts on one site from accessing data on another. For a penetration tester, an insecure CORS policy—especially when combined with Access-Control-Allow-Credentials: true—is a high-priority finding because it allows an attacker to host a malicious site that, when visited by a logged-in victim, can silently make authenticated API calls to the target server to exfiltrate private information like profile details, CSRF tokens, or internal messages.

###### Content Management System (CMS)
A Content Management System (CMS) is a software application or suite of related programs used to create, manage, and modify digital content on a website without requiring specialized technical knowledge or direct coding in HTML and CSS. Popular examples like WordPress, Joomla, and Drupal provide a graphical user interface (GUI) and a centralized database to store text, images, and user data, allowing multiple contributors to collaborate on a single codebase. From a penetration testing perspective, a CMS represents a concentrated attack surface because vulnerabilities are often found in third-party plugins and themes rather than the core software itself. Identifying the specific CMS and its version using tools like CMSeeK or WPScan is a critical first step, as it allows a tester to search for known public exploits, default administrative paths like /wp-admin/, or insecure configuration files that could lead to full site compromise.

###### Virtual host
A virtual host is a configuration on a single web server (like Apache or Nginx) that allows it to host and serve multiple different websites or domain names from a single IP address. When a browser sends an HTTP request, it includes a Host header specifying the domain it wants to reach; the server then uses this header to "route" the traffic to the specific directory and codebase associated with that site. For a penetration tester, discovering "hidden" virtual hosts is a critical task because an organization might have a secure public website on www.example.com but a completely unpatched, vulnerable development site on dev.example.com residing on the same physical server. Failing to brute-force or fuzz for these headers means you are only seeing the "default" application and missing a potentially massive internal or forgotten attack surface.

###### Git-dump
A **git-dump** refers to the automated process of reconstructing a website's entire source code and development history by downloading the contents of an exposed `/.git` directory that was accidentally left publicly accessible on a web server. Using a tool like **git-dumper**, a penetration tester can fetch internal Git objects (like commits, trees, and blobs) and the `index` file to rebuild the original folder structure, allowing them to browse the full **codebase** as if they had cloned it from a private repository. This is a critical finding because the dumped data often reveals sensitive "hardcoded" secrets—such as database passwords, **API keys**, and developer notes—contained within the version history that are not visible in the "live" version of the website.

###### GraphQL
GraphQL is a query language and runtime for **APIs** that allows clients to request exactly the data they need and nothing more, contrasting with traditional REST APIs that often return fixed, over-relying data structures. Instead of hitting multiple endpoints for different resources, a client sends a single POST request to a unique `/graphql` endpoint with a specific query defining the desired fields, which the server then resolves by fetching data from various backends or databases. For a penetration tester, GraphQL introduces unique attack vectors such as **Introspection queries**, which can reveal the entire API schema including hidden administrative functions, and **Batching attacks**, where an attacker wraps hundreds of queries into a single HTTP request to bypass rate limiting or brute-force credentials.

###### Introspection
The term introspection is a general concept in computer science and programming that refers to the ability of a program to examine its own structure, metadata, and properties at runtime. While it is a defining feature of GraphQL, where it allows a client to query the API for its own schema, the concept exists across many languages and environments, such as Python using the dir() or type() functions, Java through its Reflection API, and even in OpenID Connect for validating security tokens. In a security context, introspection is generally considered a reconnaissance goldmine because it replaces blind "black-box" guessing with a clear, machine-readable map of an application's internal logic, objects, and available functions.

###### Content Delivery Network (CDN)
A Content Delivery Network (CDN) is a geographically distributed group of caching servers that work together to provide fast delivery of internet content by minimizing the physical distance between the user and the website's hosting server. When a user requests a file, the CDN transparently intercepts the request and serves a cached version of the data from the "edge" server nearest to them, reducing latency and offloading bandwidth from the origin server. For a penetration tester, a CDN acts as a defensive shield that often masks the target's true origin IP address, hides the backend infrastructure, and provides built-in protection against DDoS attacks and common web exploits. Bypassing a CDN to find the direct IP of the origin server is a high-priority reconnaissance task, as it allows a tester to attack the web application directly without being blocked by the CDN's security rules or rate-limiting features.

###### Rate limiting
Rate limiting is a network traffic management strategy that caps the number of requests a user or IP address can make to a web application or API within a specific timeframe, such as 100 requests per minute. By enforcing these thresholds, organizations protect their infrastructure from Brute-Force attacks, Denial of Service (DoS), and automated scraping bots that could otherwise overwhelm the server's resources. For a penetration tester, identifying the presence of rate limiting is a critical step in the reconnaissance phase, as it dictates the speed and "aggression" of your automated tools; if you trigger a rate limit, the server may respond with an HTTP 429 Too Many Requests status code, temporarily block your IP, or silently drop your packets. Discovering ways to bypass these limits—such as by rotating through a pool of proxy IPs, spoofing the X-Forwarded-For header, or introducing random delays (jitter) between requests—is a common objective when attempting to perform large-scale credential stuffing or directory enumeration.

###### Netblock
In network reconnaissance, a netblock is a contiguous range of IP addresses assigned to a single organization or service provider, typically defined using **CIDR (Classless Inter-Domain Routing)** notation such as `192.168.1.0/24`. For a penetration tester, identifying an organization's netblocks is a foundational step because it defines the "scope" of the legal engagement and reveals the total infrastructure footprint, including secondary servers, development environments, and IoT devices that might not be associated with a public domain name. By querying **WHOIS** databases or using tools like `amass` and `asnlookup`, you can map out these ranges to ensure your port scanning and vulnerability assessments cover all assets owned by the target, rather than just the primary web server, which often leads to the discovery of neglected or poorly secured "shadow IT" systems.

###### Name server
A name server is a specialized server in the **Domain Name System (DNS)** that acts as a directory to translate human-readable domain names into the numerical IP addresses computers use to communicate. When you enter a URL into your browser, the request eventually reaches an **Authoritative Name Server**, which holds the "master record" (such as A, AAAA, or MX records) for that specific domain and provides the definitive IP address to the requester. For a penetration tester, identifying a target's name servers is a fundamental reconnaissance step because it reveals who manages the organization's DNS infrastructure and can lead to the discovery of additional subdomains through **Zone Transfers (AXFR)** if the servers are misconfigured. Furthermore, name servers often point to third-party providers like Cloudflare or AWS, giving you immediate insight into the security services and **CDN** protections currently shielding the target's backend.

###### GDPR redaction
GDPR redaction is the process of permanently and irreversibly removing or masking personally identifiable information (PII) from documents and datasets to protect individual privacy and ensure legal compliance. Under the General Data Protection Regulation, organizations must be able to fulfill "Right to Erasure" requests and protect sensitive data—such as names, addresses, and IP addresses—from unauthorized access or accidental disclosure. Effective redaction goes beyond simply covering text with black boxes; it requires technical methods like data masking, pseudonymization, or anonymization to ensure that the underlying metadata and searchable text are completely purged. By applying these techniques to logs, databases, and physical records, entities can share or store information safely while mitigating the risk of heavy financial penalties associated with data breaches.

###### Querying
the process of requesting, retrieving, or manipulating data from a database or storage system using specific, structured commands.

###### incremental zone transfer
Incremental Zone Transfer (IXFR), defined in RFC 1995, is an efficient DNS mechanism that allows a secondary server to update its zone data by requesting only the specific changes made since the last synchronization, rather than downloading the entire zone file as seen in a Full Zone Transfer (AXFR). When the secondary server detects a change via a serial number increase in the SOA record, it sends an IXFR request containing its current serial number; the primary server then responds with a sequence of RRsets representing the deletions and additions necessary to bring the secondary up to date. This significantly reduces network bandwidth consumption and processing overhead, particularly for large zones with frequent, minor updates.

###### Wildcard filtering
Wildcard filtering is a search and data-processing technique that uses placeholder characters, typically an asterisk (`*`) for multiple characters or a question mark (`?`) for a single character, to match patterns within strings rather than requiring exact values. In networking and security, it is often used to manage large sets of data or domains efficiently, such as blocking all subdomains of a malicious site using `*.malicious.com` or searching through log files for specific error codes using grep. While highly effective for broad discovery and automation, over-permissive wildcard filters can introduce security vulnerabilities by accidentally matching unintended targets, and leading with a wildcard in database queries often results in significant performance degradation because it prevents the system from utilizing optimized search indexes.

###### Domain Name System Security Extensions (DNSSEC)
Domain Name System Security Extensions (DNSSEC) is a suite of protocols designed to secure DNS data by adding digital signatures to existing DNS records, ensuring that the information received by a resolver is authentic and has not been tampered with during transit. By using public-key cryptography, DNSSEC allows a recursive resolver to verify a chain of trust from the root zone down to the specific domain, effectively preventing common attacks like DNS cache poisoning and man-in-the-middle interceptions. While DNSSEC provides data integrity and origin authentication, it does not provide data confidentiality or encryption for DNS queries; its primary role is to guarantee that a user seeking a specific website is not redirected to a malicious IP address by a forged response.

###### Subdomain
A subdomain is an additional part of a main domain name used to organize and navigate to different sections of a website or to separate distinct services under a single administrative umbrella. Located to the left of the root domain and separated by a dot—such as `dev` in `dev.example.com`—it acts as a pointer to a specific directory or a different server altogether. From a technical and security standpoint, subdomains are often used to isolate development environments, host mail servers, or manage localized content, but they can also expand an organization's attack surface if "shadow IT" or forgotten subdomains are left unmonitored.

###### Obscure

###### CAPTCHAs (Completely Automated Public Turing test to tell Computers and Humans Apart)
CAPTCHAs (Completely Automated Public Turing test to tell Computers and Humans Apart) are security mechanisms designed to distinguish between human users and automated bots by presenting challenges that are easy for people to solve but difficult for current artificial intelligence. They serve as a vital defense against "brute-force" attacks, spam, and unauthorized data scraping by requiring users to perform tasks such as identifying specific objects in a grid of images, typing distorted text, or simply interacting with a checkbox that analyzes behavioral patterns. While traditional text-based challenges have become less effective as machine learning improves, modern versions like Google's reCAPTCHA v3 operate invisibly in the background, assigning a risk score based on user interaction to block malicious traffic without disrupting the legitimate user experience.

###### Web interface
A web interface is a browser-based graphical user interface that allows users to interact with software, data, or network devices over the HTTP/IP protocol using standard web technologies like HTML, CSS, and JavaScript. By offloading the application logic to a remote server and using the browser as a universal interpreter, web interfaces provide cross-platform accessibility from any device without requiring local software installation. In the context of networking and security, these interfaces serve as critical management points for firewalls, routers, and cloud services, but they also represent a primary attack surface prone to vulnerabilities such as Cross-Site Scripting (XSS) and Broken Access Control if not properly hardened with input validation and encrypted sessions.

###### Anchor text
Anchor text refers to the clickable text in a hyperlink, and in the context of Google dorking, it is targeted using the `inanchor:` or `allinanchor:` operators to discover pages that are being linked to with specific keywords. Since anchor text often describes the content or purpose of the destination page more accurately than the page's own title, attackers use these dorks to identify potentially vulnerable services, such as `inanchor:"login"` or `inanchor:"admin panel"`, which might not explicitly state those terms in their headers. This technique is particularly effective for mapping internal infrastructure or finding "shadow IT" because it relies on how other pages—either internal or external—perceive and label a target URL, often revealing sensitive entry points that have been obscured from direct search results.

###### Pastebin and paste sites
Pastebin and other paste sites are online content-hosting services that allow users to store and share plain text or source code snippets through public or private URLs, originally designed to facilitate collaboration among developers. In the context of cybersecurity and Google dorking, these platforms are a goldmine for reconnaissance because they are frequently used—intentionally or accidentally—to host sensitive data such as database credentials, API keys, private configuration files, or leaked "combo lists" from data breaches. Attackers use specific dorks like `site:pastebin.com "password"` or `site:controlc.com "ssh-rsa"` to scan these repositories for exposed secrets that have been indexed by search engines. While many sites now implement automated scraping protection and content moderation, they remain a primary target for "dump" monitoring and initial access discovery during the early stages of an assessment.

###### Rainbow table
A rainbow table is a precomputed data structure used to reverse cryptographic hash functions, primarily for cracking password hashes by trading off storage space for computing speed. Unlike a simple lookup table that stores every possible password and its corresponding hash, a rainbow table uses "reduction functions" to create long chains of passwords and hashes, storing only the starting password and the final hash of each chain. When an attacker encounters a target hash, they apply the reduction and hashing sequence until they find a match in the table's "end" column, then regenerate the specific chain to pinpoint the original password. This method significantly accelerates the cracking process for unsalted hashes, though its effectiveness is completely neutralized by "salting," which adds unique, random data to each password before hashing to ensure that identical passwords produce different hash outputs.

###### TTPs (Tactics, Techniques, and Procedures)
In the context of cybersecurity, **TTPs** (Tactics, Techniques, and Procedures) describe the behavior of a threat actor and are used to characterize how they orchestrate and execute attacks. **Tactics** represent the "what" or the strategic objective of the attacker, such as Initial Access or Lateral Movement; **Techniques** describe the "how" or the specific method used to achieve a tactic, like using Spearphishing or Credential Dumping; and **Procedures** are the detailed, step-by-step sequences of actions or specific tools used in a real-world implementation. By mapping these behaviors to frameworks like **MITRE ATT&CK**, defenders can move beyond simple Indicators of Compromise (IoCs) like IP addresses or file hashes—which are easy for attackers to change—and instead focus on detecting and disrupting the fundamental patterns of operation that are much harder for an adversary to alter.

###### Single Sign-On (SSO) and Identity Provider (IdP)
Single Sign-On (SSO) is an authentication scheme that allows a user to log in with a single set of credentials to access multiple independent software systems, while an Identity Provider (IdP) is the centralized service that stores and manages those digital identities. In this architecture, when a user attempts to access a "Service Provider" (like Slack or Zoom), the request is redirected to the IdP, which authenticates the user and issues a secure token—often using protocols like SAML 2.0 or OpenID Connect—to prove the user's identity back to the service. This centralization improves security by reducing the "password fatigue" that leads to weak credentials and allows administrators to instantly revoke access across all linked applications from a single dashboard. For a network engineer or penetration tester, the IdP represents a critical "crown jewel" of infrastructure; if compromised, it grants an attacker the keys to the entire enterprise environment, making features like hardware-based MFA and conditional access policies essential for hardening the SSO flow.

###### AXFR (Asynchronous Transfer Full Range) request
An AXFR (Asynchronous Transfer Full Range) request is a specific type of DNS query used to initiate a **zone transfer**, which replicates the entire DNS database of a domain from a primary server to a secondary server. Unlike standard DNS queries that look up a single hostname, a successful AXFR request returns every resource record in the zone—including subdomains, mail servers (MX), and internal IP mappings—typically over **TCP port 53** to ensure data integrity. From a security perspective, an unrestricted AXFR request is a major vulnerability because it allows an attacker to perform "zone enumeration," instantly mapping an organization's entire internal and external infrastructure without the need for noisy brute-force or subdomain guessing. To prevent this, administrators should restrict AXFR permissions only to the IP addresses of authorized secondary name servers and implement **TSIG (Transaction Signatures)** for authenticated communication.

###### Access Control List (ACL)
An Access Control List (ACL) is a sequential collection of permit or deny statements, called Access Control Entries (ACEs), used by network devices like routers and firewalls to filter traffic based on specific criteria such as source and destination IP addresses, protocols, or port numbers. Operating primarily at Layer 3 (Network) or Layer 4 (Transport) of the OSI model, ACLs act as the first line of defense in a network by governing which packets are allowed to transit an interface or reach a sensitive segment. Standard ACLs typically filter based solely on the source IP address, while Extended ACLs provide granular control by inspecting the entire packet header, including the application-level ports (e.g., TCP 80 for HTTP). In any professional network configuration, it is critical to remember the "implicit deny" rule: if a packet does not match any specific entry in the list, it is automatically dropped by default, which ensures that only explicitly authorized traffic can pass through the gateway.
